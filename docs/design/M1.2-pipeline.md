# M1.2 — Pipeline Lead (mini-note de cadrage)

> Statut : proposition à valider · Prérequis : M1.1 clôturé + revue de santé appliquée.
> Références : [M1-conception](M1-conception.md) §4, [DOMAIN-MODEL](../architecture/DOMAIN-MODEL.md),
> ADR-0003 (events), ADR-0013 (read models). Métier en FR, code en EN (glossaire).

## 1. Objectif

Le **suivi commercial** : créer une Piste depuis une Organisation, la faire avancer dans la
machine à états, garder la trace de chaque interaction (journal), visualiser le tout en
kanban + fiche. Fin de slice : la traductrice pilote son démarchage réel dans Plume.

Hors périmètre (slices suivantes) : relances/`FollowUp` et régularité (M1.3),
génération IA (M1.4), KPIs (M1.5).

## 2. Domaine — enrichir l'agrégat `Lead`

État M0 : `LeadId`, `TenantId`, `organizationId`, `segment`, `status` + `create()`/`contact()`.

**Champs à ajouter** : `contactId?` (référence par ID — jamais l'objet), `languagePair`
(**nouveau VO `LanguagePair`** : source→cible, ex. `en→fr`), `source` (**enum `LeadSource`** :
`DIRECT` | `REFERRAL` | `JOB_BOARD` | `OTHER`), `priority` (**enum `Priority`** : `LOW` |
`MEDIUM` | `HIGH`), horodatages (`createdAt`, `lastContactedAt?`, `lastReplyAt?`).

**Méthodes + events (un event par fait métier, tous horodatés par `Clock`)** :

| Méthode | Transition | Event |
|---|---|---|
| `create()` *(existe)* | → `TO_CONTACT` | `LeadCreated` |
| `contact()` *(existe)* | `TO_CONTACT`/`FOLLOWED_UP` → `CONTACTED` | `LeadContacted` |
| `recordReply()` | actif → `IN_DISCUSSION` | `ReplyReceived` |
| `moveToSampleTest()` | `IN_DISCUSSION` → `SAMPLE_TEST` | `LeadMovedToSampleTest` |
| `markWon()` / `markLost()` | actifs → `WON`/`LOST` | `LeadWon` / `LeadLost` |
| `pause()` / `resume()` | actif (sauf `SAMPLE_TEST`) ↔ `PAUSED` | `LeadPaused` / `LeadResumed` |
| `addNote(text)` | (aucune) | `NoteAdded` (le texte vit dans l'event → journal, pas dans l'agrégat) |

**Invariants** : transitions limitées au graphe (`IllegalStatusTransition`) ; états terminaux
figés ; **le démarchage d'une organisation `doNotContact` est interdit** (création + `contact()`) ;
tenant cohérent Piste/Organisation.

**Frontière de contexte** : Prospecting vérifie l'Organisation via un **port**
`App\Prospecting\Application\OrganizationGateway` (`exists()`, `isContactAllowed()`, `nameOf()`)
implémenté en Infrastructure sur le read model Directory. Jamais d'accès direct à l'agrégat.

## 3. Journal d'interactions — première vraie projection

Table append-only `interaction` (`id`, `tenant_id`, `lead_id`, `type`, `payload` JSONB,
`occurred_on`) alimentée **en asynchrone** par les domain events (worker Messenger + outbox
déjà câblés).

Points de conception (c'est le moment de faire ça bien) :
- **Events enrichis** : chaque event Lead porte `tenantId` + les données d'affichage
  nécessaires (fat event) → le consumer écrit sans recharger d'agrégat ni contexte tenant.
- **Idempotence** : `DomainEvent` gagne un `eventId` (UUID) ; `interaction.event_id` UNIQUE —
  un retry Messenger ne duplique jamais une ligne.
- Timeline de la fiche = lecture SQL de `interaction` (port `LeadTimeline`, fail-closed tenant).
- Le worker (profil `full`) devient nécessaire au quotidien → documenté README ; en test
  fonctionnel, consommation synchrone du transport (`messenger:consume --limit`).

## 4. Application & API

**Commands** : `CreateLead`, `ContactLead` *(existe — à compléter : gateway doNotContact)*,
`RecordReply`, `MoveToSampleTest`, `MarkLeadWon`, `MarkLeadLost`, `PauseLead`, `ResumeLead`,
`AddLeadNote`. **Queries** : `ListLeads` (filtres statut/priorité/segment + pagination —
le kanban groupe côté client, volumes faibles), `GetLead`, `GetLeadTimeline`.
Read models : `LeadView`/`LeadPage` (port `LeadSearch`, pattern ADR-0013).

**API orientée cas d'usage** (pas de PATCH fourre-tout sur le statut) :
- `POST /api/v1/leads` (organizationId, contactId?, languagePair, source, priority, segment)
- `GET /api/v1/leads` (+ filtres), `GET /api/v1/leads/{id}`, `GET /api/v1/leads/{id}/interactions`
- Transitions : `POST /api/v1/leads/{id}/contact` · `/reply` · `/sample-test` · `/win` ·
  `/lose` · `/pause` · `/resume` · `/notes`
- Transition interdite → **409** (mapping `Conflict` existant). Contrats Assert complets + openapi.

## 5. Frontend

- **Pistes (kanban)** : colonnes = statuts, cartes (organisation, paire de langues, priorité,
  segment, dernière activité) ; **actions de transition par boutons sur la carte/fiche en V1**,
  drag-and-drop en itération suivante (a11y d'abord — cf. micro-décision 3).
- **Fiche Piste** : en-tête (org → lien Répertoire, statut, priorité), actions contextuelles
  (seules les transitions légales sont proposées), **timeline** du journal, ajout de note.
- **Création** : depuis la fiche Organisation (« Créer une piste ») et depuis le kanban.
- Nav « Pistes » activée. i18n FR/EN, toasts, confirmations sur `markLost` (destructif métier).
- E2E étendu : créer piste → contacter → répondre → gagner (+ garde console).

## 6. Tests (pyramide, comme M1.1)

Domaine : machine à états **exhaustive** (toutes transitions légales/illégales), invariants.
Application : handlers in-memory (gateway mockée : org inexistante, doNotContact).
Fonctionnel : parcours API complet, isolation tenant, 409 sur transitions illégales,
projection `interaction` (consommation synchrone du transport), idempotence.
E2E : parcours kanban.

## 7. Micro-décisions à valider

1. **Une seule piste active par organisation ?** (recommandé : **oui** — deux démarchages
   parallèles sur la même cible n'ont pas de sens ; contrainte à la création, `Conflict` sinon)
2. **`estimatedValue` (VO `Rate`)** : différé (recommandé — YAGNI tant que le dashboard M1.5
   n'agrège pas de montants) ?
3. **Kanban V1 sans drag-and-drop** (boutons de transition accessibles), DnD en itération 2 ?
4. **Journal asynchrone** (worker + fat events + idempotence, cf. §3) — ou projection
   synchrone jetable en attendant M1.3 ? (recommandé : **asynchrone**, l'infra existe et
   M1.3/M1.5 en dépendront)
5. Valeurs de `LeadSource` : `DIRECT` | `REFERRAL` | `JOB_BOARD` | `OTHER` — OK ?

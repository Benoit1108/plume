# V2.0 — Prérequis & bascule multi-tenant (note de conception)

- **Statut** : proposé (à valider avant exécution)
- **Amont** : [cadrage V2](./V2-cadrage.md). V2.0 = **rendre la plateforme prête pour une ouverture
  publique multi-comptes**, avant l'inscription (V2.1). Modèle acté : **1 compte = 1 traductrice**.
- **Principe** : la V1 reste utilisable ; livraison par tranches, CI verte, revue en fin de jalon.

## Ce que le choix « 1 compte = 1 traductrice » simplifie

- **`app_user` reste HORS RLS** (et c'est le bon choix, pas une dette) :
  - Le provider Symfony `app_users` charge l'utilisateur **par email, AVANT que le tenant soit
    connu** (au login `json_login`, et au rechargement stateless de chaque requête, avant que
    `TenantContextListener` n'active le tenant sur `on_jwt_authenticated`). Sous RLS, ce lookup
    renverrait 0 ligne → **auth cassée**.
  - En 1=1, il n'existe **aucune** fonctionnalité de listing d'utilisateurs (pas d'équipe) : la
    seule requête sur `app_user` est « trouver par email » (auth) — pas de surface d'énumération
    cross-tenant exposée. `email` est unique global → pas d'ambiguïté.
  - → **Décision : app_user reste hors RLS, documentée dans ADR-0023 §4.** À rouvrir UNIQUEMENT si
    le workspace multi-utilisateurs arrive (alors : policy autorisant le lookup par email hors tenant,
    ou provider dédié). Pas de travail V2.0 ici.
- Pas de gestion de rôles/permissions ni d'UI d'équipe → V2.1 (comptes) reste léger.

## Tranches V2.0 (engineering — exécutables)

### V2.0-a — RGPD : export & suppression de compte *(prérequis LÉGAL d'une ouverture publique)*
Un SaaS public doit offrir portabilité + droit à l'effacement.
- **Export** des données d'un tenant : endpoint authentifié `GET /api/v1/account/export` → archive
  (JSON) de tout le tenant (organisations, contacts, pistes, journal, brouillons, gabarits, profil,
  flux, candidats). Lecture via les read models existants (déjà scopés tenant).
- **Suppression de compte** : `DELETE /api/v1/account` → purge de TOUT le tenant (toutes les tables
  métier + app_user + refresh_tokens + boîte connectée + tokens chiffrés + brut d'annonces), avec
  confirmation forte (ré-auth / mot de passe). Révoque l'OAuth chez le fournisseur au passage.
  Transactionnel + tracé (log d'audit).
- **Registre de traitement** (doc) + **DPA** (doc) : sous-traitants = Anthropic (IA), Google/Microsoft
  (mail), + l'hébergeur. Politique de rétention (journal, brut d'alertes, outbox `failed`).

### V2.0-b — Robustesse à l'échelle (dettes ADR-0022 §5, §4, §3)
- **§5 — Isolation de charge inter-contextes** : aujourd'hui **un seul** transport `async` + un seul
  worker. À l'échelle multi-tenant, une relève RSS/email lente peut retarder les projections/policies
  de tous. → **séparer les transports Messenger** (ex. `async` pour les events/projections légers ;
  `io` dédié pour les commandes à I/O réseau — poll/fetch), workers distincts. Impacte compose + prod.
- **§4 — Tables hors ORM** (`interaction`, `raw_alert`, `messenger_messages`) + `schema_filter`
  denylist : **documenter leur schéma en un lieu unique** (un fichier + les migrations font foi) ;
  vérifier que la denylist ne masque pas un DROP au diff. Faible effort.
- **§3 — Patrons d'adaptateurs** : la source de démo est neutralisée en prod (fait). Reste
  l'hétérogénéité Selector (Drafting) vs registres (Mailbox). **À évaluer** : harmoniser en registres,
  ou acter que c'est acceptable. Décision, pas forcément du code.

### V2.0-c — Préparation hébergement AGNOSTIQUE *(sans trancher VPS vs PaaS)*
Abstraire ce qui diffère, pour que le choix se fasse au 1er déploiement sans dette.
- **Store des rate-limiters / login-throttling** : aujourd'hui cache filesystem (mono-hôte). →
  configurable (Redis) via env, pour un déploiement multi-instances éventuel.
- **Worker-mode FrankenPHP** : SI activé en prod (perf), le reset tenant `kernel.request` (défensif,
  différé en pré-V2) devient nécessaire. → l'ajouter, conditionné/inoffensif en mode classique.
- **Checklist de déploiement** : surcharge obligatoire `APP_SECRET`, `JWT_PASSPHRASE`,
  `APP_DB_PASSWORD`, `MAILBOX_ENCRYPTION_KEY`, `TRUSTED_PROXIES` ; sauvegardes DB chiffrées ;
  rotation clés. (Doc + garde fail-fast là où c'est possible.)

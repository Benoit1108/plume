# M1.5 — Tableau de bord (mini-note de cadrage)

> Statut : **livré** (2026-07-14, 6 micro-décisions validées) · Prérequis : M1.4 clôturé.
> Références : [M1-conception](M1-conception.md) §2/§9, ADR-0013 (read models — query directe,
> seul le journal est projeté). Métier en FR, code en EN (glossaire : Tableau de bord = `Dashboard`).

## 1. Objectif

Répondre en un écran aux trois questions de pilotage : **ma prospection paie-t-elle ?**
(taux de réponse, conversion), **où en est mon pipeline ?** (répartition par statut),
**suis-je régulière ?** (activité des dernières semaines vs objectif). Le tableau de bord
**oriente** aussi : les taux par segment montrent où concentrer l'effort (édition vs A/V
vs technique).

« Aujourd'hui » reste l'écran d'action (quoi faire maintenant) ; le tableau de bord est
l'écran de recul (comment ça se passe). Pas de nouveau concept métier : **slice 100 % lecture**
— aucun agrégat, aucun event, aucune migration.

## 2. Contenu (V1)

1. **KPIs globaux** (cartes) :
   - **Taux de réponse** = pistes ayant reçu ≥ 1 réponse / pistes contactées (cumul, par
     piste — pas par message).
   - **Conversion** = gagnées / décidées (gagnées + perdues) — mesurer sur les pistes
     *décidées* évite d'écraser le taux pendant que le pipeline est plein d'en-cours.
   - **Actes ce mois-ci** (contacts + relances, journal `contacted`/`followed_up` —
     mêmes types que la progression M1.3) + total pistes actives.
2. **Pipeline** : répartition des pistes par statut (barre segmentée, mêmes couleurs que
   le kanban), pistes actives vs terminées.
3. **Activité hebdomadaire** : 8 dernières semaines (semaine ISO, fuseau du profil),
   barres des actes avec la ligne d'objectif — l'historique de la régularité, en
   prolongement du widget « Aujourd'hui ».
4. **Par segment** (tableau) : contactées / réponses (taux) / gagnées, par segment —
   le « où ça mord » de la traductrice.

Hors périmètre (reportés, **notés en ROADMAP § M2 — tableau de bord enrichi**) : délais
moyens de réponse, valeur estimée (`estimatedValue` différé depuis M1.2), filtres de période,
export. Les taux sont globaux (tout l'historique) : au volume d'une prospection individuelle,
les fenêtres glissantes seraient du bruit en V1.

## 3. Architecture

- **Read model** `Dashboard` (port, Prospecting\Application) → `DashboardView` immuable :
  `responseRate` (num/den exposés, pas seulement le ratio — l'UI affiche « 4 / 12 »),
  `conversion`, `pipeline[]` (statut, count), `weeklyActivity[]` (weekStart, actes),
  `weeklyTarget`, `segments[]` (segment, contacted, replied, won), `activeLeads`,
  `outreachThisMonth`.
- **Implémentation DBAL** `DoctrineDashboard` : agrégations SQL sur `lead` + `interaction`,
  tenant explicite **fail-closed** (ADR-0013), semaine ISO dans le fuseau du profil
  (même approche que `DoctrineTodayBoard`). Taux calculés sur le **journal** (`contacted`,
  `reply`, `won` par `lead_id` distinct) : le journal est la source de vérité de l'activité,
  et une piste gagnée puis archivée continue de compter dans l'historique.
- **API** : `GET /api/v1/dashboard` (resource singleton lecture seule, provider → query bus,
  pattern `/today`). openapi régénéré **après** `cache:clear` (piège M1.4).
- **Front** : page `/dashboard`, entrée de nav « Tableau de bord » (entre Aujourd'hui et
  Pistes). Graphes **maison** (barres CSS/SVG légères, thème Encre violette, sombre/clair) :
  pas de lib de charts pour 4 visualisations simples — zéro dépendance, a11y maîtrisée
  (valeurs en texte, `aria-label`, tableau accessible en repli).

## 4. Micro-décisions à valider

1. **Conversion = gagnées / décidées** (gagnées + perdues), plutôt que gagnées / contactées ?
   *(recommandé : oui — plus honnête tant que le pipeline est jeune ; les deux dénominateurs
   sont affichés en clair)*
2. **Taux calculés sur le journal** (piste comptée « contactée » si un acte `contacted`
   existe), pas sur le statut courant — l'historique survit aux transitions ? *(recommandé : oui)*
3. **8 semaines** d'activité hebdomadaire, objectif courant en ligne de référence ?
   *(recommandé : oui — assez pour voir une tendance, tient sur mobile)*
4. **Graphes maison** (CSS/SVG + Tailwind) plutôt qu'une lib de charts ? *(recommandé : oui —
   4 visuels simples, design system respecté, pas de dépendance à auditer)*
5. **`/dashboard` en 2ᵉ entrée de nav**, « Aujourd'hui » reste l'accueil (`/` → `/today`) ?
   *(recommandé : oui — l'action d'abord, le pilotage à un clic)*
6. Slice **lecture seule** : pas de projection dédiée (queries directes, décision M1 n°3 /
   ADR-0013), pas de cache — on optimisera si les agrégations pèsent un jour ? *(recommandé : oui)*

# M3.1 — Ingestion RSS (note de cadrage)

> Sous-jalon de M3 (voir `M3-conception.md`, décisions **D1→D7 validées**). Prérequis :
> **M3.0 clôturé + revue M3.0 remédiée** (socle Sourcing, file « À trier », dédoublonnage).
> Métier en FR, code en EN. ADR-0020 (contexte Sourcing) / ADR-0021 (dédoublonnage) font foi.

## 1. Objectif

Aujourd'hui la file « À trier » ne se remplit que par la **saisie manuelle** (`Source::MANUAL`).
M3.1 lui donne sa **première source automatique** : un **flux RSS** (ProZ & co) est **relevé**,
chaque item devient une `CandidateLead` `PENDING` — dédoublonnée — et le **contenu brut** est
conservé pour audit/reprocessing. La boucle **RSS → dédoublonnage → « À trier »** tourne de bout
en bout, testée sans réseau.

## 2. Découpage retenu — **M3.1a « moteur d'abord »**

Cette passe livre le **moteur d'ingestion** ; la **gestion des flux** (agrégat `AlertFeed`,
CRUD, écran Réglages « Sources ») et le **Scheduler automatique** (relève périodique, fan-out
tous tenants) sont reportés à **M3.1b**.

| Dans M3.1a | Reporté en M3.1b |
|---|---|
| Port `AlertSource` + `RssAlertSource` (HttpClient réel) + `FakeAlertSource` (démo, défaut) | Agrégat `AlertFeed` (URL/source/actif) persisté par tenant |
| Entité support `RawAlert` (brut conservé) + `rawRef` sur `CandidateLead` | Écran Réglages « Sources » (CRUD des flux) |
| Déclenchement **manuel** `POST /sources/poll` (tenant **courant**) | `Scheduler` auto (5 min) + fan-out par tenant (`PollAllSourcesTick`) |
| Bouton « Relever les annonces » sur « À trier » + E2E de la boucle | Purge planifiée du brut `RawAlert` (D6, J+30) |

**Pourquoi ce découpage** : dé-risquer le cœur (parsing RSS robuste, conservation du brut,
dédoublonnage réel avec l'`externalId`=guid) avant d'investir dans la config UI et l'automatisme.
Chaque passe est un incrément CI-vert livrable.

## 3. Modèle

- **`RawAlert`** (support, **hors** agrégat métier — patron de la projection `interaction`) :
  `RawAlertId`, `TenantId`, `Source`, `payload` (item RSS brut), `fetchedAt`. Écrit en **DBAL**
  (tenant explicite), table non mappée ORM (ajoutée au `schema_filter`). Pas de domain event.
- **`CandidateLead`** gagne `rawRef` (nullable) : référence vers le `RawAlert` d'origine
  (null pour la saisie `MANUAL`). Champ ajouté en fin de signature `ingest()` (défaut null).
- **`ParsedAlert`** (DTO Application) : `source`, `title`, `organizationName?`, `languagePair?`,
  `url?`, `excerpt?`, `externalId?` (guid RSS → dédoublonnage stable), `postedAt?`, `rawPayload`.

## 4. Flux (synchrone, comme la relève manuelle M2)

Les commandes tournent **sync** (seuls les `DomainEvent` sont routés async). Le geste manuel fait
donc son I/O réseau dans la requête — assumé pour un « relever maintenant » (identique à
`POST /mailbox/fetch-replies`). L'automatisme périodique (worker) viendra en M3.1b.

```
POST /sources/poll (tenant courant)
  └─ PollAlertSource(tenantId)                       [command.bus, 1 transaction]
       └─ AlertSource::fetch(url) par flux actif   → iterable<ParsedAlert>   (Fake en repli si aucun flux)
            └─ pour chaque item : IngestCandidate(...rawPayload)   [dispatch imbriqué]
                 ├─ dédoublonnage (dedupHash sur source+externalId) → no-op si déjà vue
                 └─ RawAlert conservé + CandidateLead PENDING (rawRef)
```

- **Best-effort au parsing** : `RssAlertSource` **saute** un item malformé (titre vide, XML
  cassé) sans interrompre le lot ; bornage strict des champs (réutilise les longueurs des inputs).
- **Sélection de la source** *(livré M3.1b — remplace le mécanisme par env de ce cadrage M3.1a)* :
  `PollAlertSource` itère les **flux `AlertFeed` actifs du tenant** et appelle `RssAlertSource::fetch(url)`
  par flux ; si le tenant n'a aucun flux actif, repli sur `FakeAlertSource` (démo, zéro réseau —
  dev/CI/E2E). L'env `SOURCING_RSS_FEED_URL` et `AlertSourceFactory` du cadrage initial n'ont pas
  été retenus. Cf. `DOMAIN-MODEL.md` §Sourcing pour le livré.

## 5. API & Front

- **API** : `POST /sources/poll` → 202 (relève lancée pour le tenant courant), `output: false`.
  Rate limiting réutilisé si pertinent.
- **Front** : bouton **« Relever les annonces »** sur « À trier » → toast → rafraîchit la file.

## 6. Sécurité / RGPD

- **Isolation tenant fail-closed** sur `raw_alert` (tenant explicite en DBAL) et l'ingestion.
- `RssAlertSource` ne lit qu'une **URL de flux configurée** (pas de balayage). Le canal **email**
  (élargissement cadré de la relève, label dédié) reste **M3.2**.
- **Rétention du brut** (`RawAlert`) : conservé en M3.1a ; **purge planifiée D6 (J+30) reportée
  M3.1b** — tracé ici pour ne pas l'oublier.

## 7. Tests

- **Domaine** : `RawAlert` (capture), `CandidateLead` avec `rawRef`.
- **Application** : `PollAlertSourceHandler` (source stub 2 items → 2 candidates + 2 `RawAlert`,
  re-poll = dédoublonné) ; `IngestCandidateHandler` conserve le brut + `rawRef`.
- **Parser (infra, fixtures figées, aucun réseau)** : `RssAlertSource` sur un flux RSS enregistré
  → `ParsedAlert[]` corrects, item malformé **ignoré** (MockHttpClient — patron adaptateurs M2).
- **Fonctionnel** : `POST /sources/poll` (FakeAlertSource) → 202 puis la file contient les items,
  **isolée par tenant** ; re-poll idempotent (pas de doublon).
- **E2E** : boucle complète (relever → candidates apparaissent → accepter/rejeter → file vide).

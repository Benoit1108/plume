# M3.2 — Alertes email (note de cadrage)

> Dernier sous-jalon de M3 (voir `M3-conception.md`, D2 validée). Prérequis : M3.1 clôturé.
> Métier en FR, code en EN. Approche retenue : **plomberie + parser générique** (les parsers fins
> par fournisseur seront affinés plus tard avec de vrais emails).

## 1. Objectif

Certaines sources n'ont pas de flux RSS (LinkedIn notamment). M3.2 fait entrer leurs **emails
d'alerte** dans « À trier » : la Passerelle email lit un **label dédié** que la Traductrice pose
(« Plume/Alertes »), en extrait des annonces, et Sourcing les ingère — même file, même tri.

## 2. Périmètre retenu (plomberie + parser générique)

- **Passerelle email** lit les emails sous un **label dédié** (élargissement **cadré** de la
  relève, ADR-0017 amendé : on ne lit QUE ce label, jamais toute la boîte) et **notifie** Sourcing
  par un event applicatif `AlertEmailReceived` (langage publié) — jamais d'appel direct.
- **Sourcing** consomme l'event, **parse** l'email (générique : 1 annonce = 1 email ; provenance
  déduite du **domaine de l'expéditeur**) et dispatche `IngestCandidate` (dédoublonnage par
  l'identifiant de message ; email brut conservé en `RawAlert`).
- Déclenchement **automatique** (Scheduler, fan-out par boîte connectée).

**Livré maintenant** : le port `AlertEmailFetcher` + un `FakeAlertEmailFetcher` (démo, sans
réseau — chemin testé), l'event, le parser générique, la politique cross-contexte, la commande de
relève + le Scheduler. **Suivi (avec de vrais emails)** : adaptateurs réels Gmail/Outlook de
lecture du label + parsers fins par fournisseur (ProZ/TranslatorsCafe/LinkedIn).

## 3. Flux

```
Scheduler (FetchAllAlertEmailsTick) → par boîte CONNECTED : FetchAlertEmails(tenant)   [Mailbox]
  └─ AlertEmailFetcher::fetch(refreshToken, « Plume/Alertes »)  → emails du label
       └─ publish AlertEmailReceived(from, subject, body, externalId)                  [event.bus]
            └─ IngestAlertEmailOnAlertEmailReceived (Sourcing, tenant réactivé)        [policy]
                 └─ AlertEmailParser::parse(...) → IngestCandidate(...)
                      └─ dédoublonnage (externalId = id de message) + RawAlert conservé
```

## 4. Sécurité / RGPD (le point sensible)

- **Minimisation** : lecture **du seul label** « Plume/Alertes » posé par la Traductrice —
  jamais un balayage de la boîte. ADR-0017 amendé en ce sens.
- Un email d'alerte peut contenir des données personnelles → mêmes gardes que le Répertoire
  (jamais de contact importé sans passer par le tri humain) ; email brut conservé puis **purgé**
  avec la candidate (D6, déjà en place M3.1b).
- Isolation tenant fail-closed ; échec de relève = no-op silencieux (canal secondaire), le
  Scheduler repasse.

## 5. Tests

- **Parser** (unité) : provenance par domaine (`proz.com`→PROZ, `linkedin.com`→LINKEDIN,
  `translatorscafe.com`→TRANSLATORSCAFE, sinon MANUAL), 1 candidate/email, sujet vide ignoré,
  extraction du 1er lien.
- **Relève** (unité Mailbox) : boîte non connectée → no-op ; sinon un `AlertEmailReceived` par email.
- **Bout-en-bout** (fonctionnel) : publier `AlertEmailReceived` → une candidate apparaît dans la
  file (isolation tenant).

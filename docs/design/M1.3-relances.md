# M1.3 — Relances & régularité (mini-note de cadrage)

> Statut : **livré** (2026-07-13, micro-décisions validées) · Prérequis : M1.2 clôturé (journal d'interactions opérationnel).
> Références : [M1-conception](M1-conception.md) §4-5, [DOMAIN-MODEL](../architecture/DOMAIN-MODEL.md),
> ADR-0003 (events), ADR-0013 (read models). Métier en FR, code en EN (glossaire).

## 1. Objectif

Que **rien ne file entre les mailles** : chaque piste contactée porte sa prochaine relance,
l'écran « **Aujourd'hui** » dit quoi faire en ouvrant l'app, et l'**objectif hebdomadaire**
(+ série) entretient la régularité — la promesse centrale de Plume pour un temps fragmenté.

Hors périmètre : envoi réel d'emails (M2 — en M1 on *rappelle*, on n'envoie pas),
génération du message de relance (M1.4), KPIs complets (M1.5), notifications push/mail.

## 2. Domaine — la Relance vit dans l'agrégat `Lead`

**Entité `FollowUp`** (dans l'agrégat, collection JSONB — pattern ADR-0012) :
`FollowUpId`, `dueAt` (date), `label?`, `status` (`PENDING` | `DONE` | `CANCELLED`),
horodatages. **Une seule relance `PENDING` à la fois par piste** (décision n°2).

**Dénormalisé sur la ligne `lead`** (pour la requête « dues aujourd'hui ») :
`next_follow_up_at` + `next_follow_up_label` — maintenus par l'agrégat, jamais à la main.

**Cadence par défaut** (constante de domaine `FollowUpCadence::DEFAULT = [J+7, J+21, J+45]`) :
- `contact()` **auto-planifie** la 1ʳᵉ relance à J+7 (décision n°1) ;
- `recordFollowUp()` (« j'ai relancé ») : passe la relance en `DONE`, transition
  `CONTACTED|FOLLOWED_UP → FOLLOWED_UP`, auto-planifie la suivante (J+21 puis J+45,
  puis plus rien — au-delà, à la main) ;
- `scheduleFollowUp(dueAt, label?)` : planifie/replanifie manuellement (remplace la PENDING) ;
- `cancelFollowUp()` : annulation manuelle.

**Invariants (DOMAIN-MODEL §invariants 2-4)** :
- `recordReply()` **annule la relance en attente** (la discussion est ouverte, on ne relance plus) ;
- `markWon()/markLost()/pause()` annulent aussi la PENDING (terminal/pause = pas de rappel) —
  `resume()` ne replanifie pas seul (l'utilisatrice décide) ;
- pas de relance planifiable sur un statut terminal ou `PAUSED` ;
- `dueAt` dans le passé lointain refusé (`InvalidValue`) ; aujourd'hui OK.

**Events** (riches, comme M1.2 → journal) : `FollowUpScheduled(dueAt, label?, auto)`,
`FollowUpSent` (relance faite — nom du glossaire, l'« envoi » réel arrive en M2),
`FollowUpCancelled(reason: REPLY | TERMINAL | PAUSED | MANUAL)`.
Types de journal : `follow_up_scheduled`, `followed_up`, `follow_up_cancelled`.

## 3. Régularité — objectif hebdo & série

- **`Profile`** — premier agrégat du contexte **Account** : `tenantId`, `weeklyGoal`
  (défaut **5**/semaine), timezone (défaut `Europe/Paris`, décision n°4). Créé
  paresseusement au premier accès. (Le profil complet — bio, tarifs, signature — arrive en M1.4.)
- **Progression** = requête sur le journal `interaction` (il paie déjà) :
  `COUNT(type IN ('contacted','followed_up'))` sur la semaine courante (lundi 00:00 TZ
  utilisateur → UTC). Un acte de démarchage = contact **ou** relance faite (décision n°3).
- **Série (streak)** = nombre de semaines consécutives (en remontant, semaine courante
  incluse si déjà atteinte) où `COUNT >= weeklyGoal`. Calcul SQL à la volée (volumes
  faibles, pas de projection dédiée en V1 — même logique qu'ADR-0013).

## 4. « Aujourd'hui » — l'écran d'entrée

**Read model `TodayView`** (port `TodayBoard`, fail-closed tenant) :
- `followUpsDue` : pistes actives dont `next_follow_up_at <= fin de journée` (tri : échéance
  puis priorité) — les retards remontent en premier ;
- `toContact` : pistes `TO_CONTACT` (tri : priorité puis ancienneté) ;
- `weeklyProgress` : `{ target, done, streak }`.

**API** : `GET /api/v1/today` (une seule ressource lecture) ·
relances : `POST /api/v1/leads/{id}/follow-up` (relance faite),
`POST /api/v1/leads/{id}/schedule-follow-up` `{dueAt, label?}`,
`DELETE /api/v1/leads/{id}/follow-up` (annuler la planifiée) ·
profil : `GET /api/v1/profile`, `PATCH /api/v1/profile` `{weeklyGoal}`.

Pas de tâche Scheduler nécessaire en M1.3 : « dû » est une lecture sur `next_follow_up_at`
(pas d'envoi, pas de notification). Le scheduler existant reste pour la purge des tokens.

## 5. Frontend

- **Page « Aujourd'hui »** (`/today`, nav enfin complète) — **nouvel écran d'accueil**
  (`/` y atterrit, décision n°5) : widget objectif hebdo (barre de progression + série 🔥),
  section **Relances dues** (carte : organisation, échéance — retard en rouge —, actions
  rapides « Relance faite » / « Replanifier »), section **À contacter** (action « Contacter »).
- **Fiche piste** : bloc « Prochaine relance » (date + libellé, replanifier/annuler),
  actions `follow-up` dans `allowedActions` ; la timeline affiche les événements de relance.
- **Kanban** : colonne `FOLLOWED_UP` activée (elle prend vie avec les relances).
- **Réglages minimal** : l'objectif hebdo modifiable depuis le widget (popover/modale).
- i18n FR/EN, toasts, E2E : contacter → relance auto J+7 → replanifier à aujourd'hui →
  visible dans « Aujourd'hui » → « Relance faite » → progression +1.

## 6. Tests (pyramide)

Domaine : cadence auto (J+7/21/45 puis arrêt), annulation sur réponse/terminal/pause,
unicité PENDING, refus sur terminal, dueAt passé. Application : handlers (planifier,
relancer, annuler) + Profile. Fonctionnel : `/today` (dues + à contacter + progression),
calcul de série multi-semaines, isolation tenant, 409/422. E2E : parcours ci-dessus.

## 7. Micro-décisions à valider

1. **Auto-planification** à la cadence J+7 / J+21 / J+45 après `contact()` et chaque relance
   faite (date toujours modifiable) — ou planification 100 % manuelle ?
   *(recommandé : auto — c'est la promesse « rien ne file »)*
2. **Une seule relance en attente par piste** (simplifie « pas deux à la même échéance ») ?
   *(recommandé : oui en V1)*
3. La progression hebdo compte **contacts + relances faites** (actes de démarchage) ?
   *(recommandé : oui — sinon relancer « ne compte pas » et c'est démotivant)*
4. **Timezone `Europe/Paris` par défaut** en V1 (champ profil éditable plus tard) ?
   *(recommandé : oui)*
5. **`/` atterrit sur « Aujourd'hui »** (au lieu du Répertoire) ?
   *(recommandé : oui — c'est l'écran du quotidien)*
6. **Objectif hebdo par défaut : 5** actes/semaine, modifiable dans le widget ?

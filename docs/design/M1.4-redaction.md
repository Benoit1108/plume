# M1.4 — Rédaction assistée (mini-note de cadrage)

> Statut : proposition à valider · Prérequis : M1.3 clôturé (pipeline + relances + profil naissant).
> Références : [M1-conception](M1-conception.md) §6, ADR-0003 (events), ADR-0011 (locale UI ≠
> langue cible), ADR-0013 (read models). Métier en FR, code en EN (glossaire).

## 1. Objectif

Réduire le coût d'écriture de chaque démarchage : générer un **brouillon** de mail de
candidature / lettre de motivation / relance, personnalisé (profil + cible + segment +
langue cible), **toujours relu et édité** par la traductrice (*draft-first*, jamais d'envoi).
En M1 la sortie est un **copier-coller** vers son webmail — l'envoi réel arrive en M2.

Hors périmètre : envoi (M2), réécriture itérative type chat (V2 — un bouton « régénérer »
suffit en V1), tarifs dans le profil (différé, micro-décision n°3).

## 2. Coût & stratégie d'adaptateur (décision structurante)

L'API Anthropic est **payante à l'usage** (aucune offre gratuite durable) : ~0,5 à 2 centimes
par génération selon le modèle — négligeable en usage réel, mais **jamais requis en dev** :

- Port `MessageGenerator` (Application, contexte Drafting) — le domaine ignore « Claude ».
- **`CannedMessageGenerator`** (Infrastructure) : gabarit local déterministe, utilisé quand
  `ANTHROPIC_API_KEY` est absente → dev, tests, CI, E2E **à coût zéro**.
- **`ClaudeMessageGenerator`** (Infrastructure, ACL) : API Messages Anthropic, modèle piloté
  par env (`DRAFTING_MODEL`, défaut Sonnet — la qualité rédactionnelle est la vitrine du
  produit), clé API en secret d'env (jamais commitée — gitleaks veille), timeout + longueur
  de sortie bornée.
- Sélection par config d'env (compilation du conteneur) : clé présente → Claude, sinon canned.

## 3. Domaine — contexte Drafting (naissance)

**Agrégat `Draft`** (brouillon) : `DraftId`, `TenantId`, `leadId` (référence par ID),
`type` (`DraftType` : `APPLICATION_EMAIL` | `COVER_LETTER` | `FOLLOW_UP_EMAIL`),
`targetLanguage` (`LanguageCode` — la langue du prospect, pas celle de l'UI),
`subject?`, `body`, `status` (`DraftStatus` : `GENERATING` | `READY` | `FAILED`),
horodatages. Méthodes : `requestGeneration()` (création), `complete(subject, body)`,
`fail(reason)`, `edit(subject, body)` (READY uniquement), events riches
(`DraftRequested`, `DraftGenerated`, `DraftFailed`, `DraftEdited`) → journal (`draft_generated`).

**Agrégat `Template`** (gabarit) : `TemplateId`, `TenantId`, `name`, `type`, `segment`,
`language`, `subject?`, `body` (variables `{{organisation}}`, `{{contact}}`, `{{signature}}`…).
CRUD simple + **3 gabarits seedés** à la première utilisation (candidature édition FR,
candidature A/V EN, relance FR) — éditables/supprimables.

**Extension `Profile`** (Account) : `bio?` (présentation courte), `specialties?` (texte libre),
`signature?` — la matière première du prompt. PATCH /profile étendu.

**Invariants** : pas de génération pour une organisation `doNotContact` (garde RGPD via
`OrganizationGateway`, comme M1.2) ; `edit()` refusé tant que `GENERATING` ; un brouillon
appartient à une piste du même tenant.

## 4. Flux de génération (asynchrone — l'infra worker existe)

1. `POST /api/v1/leads/{id}/drafts` `{type, targetLanguage, templateId?}` →
   command `GenerateDraft` : gardes (piste, RGPD), crée le `Draft` en `GENERATING`,
   publie `DraftRequested` (event **routé async**).
2. Le worker consomme `DraftRequested` → handler d'Infrastructure : assemble le
   `DraftPrompt` (profil + organisation/contact + segment + statut de la piste + gabarit
   éventuel + langue cible), appelle le **port**, puis command `CompleteDraft` / `FailDraft`.
3. Front : la fiche piste liste les brouillons (statut visible), rafraîchissement de
   rattrapage (pattern M1.2) ; une fois `READY` → éditeur (sujet + corps), boutons
   **Copier** (presse-papier), **Régénérer**, **Supprimer**.

Échec API (panne, quota) → `FAILED` avec raison affichable + retry Messenger standard ;
jamais de message d'erreur interne exposé.

## 5. API

- `POST /api/v1/leads/{id}/drafts` (génère) · `GET /api/v1/leads/{id}/drafts` ·
  `GET /api/v1/drafts/{id}` · `PATCH /api/v1/drafts/{id}` (sujet/corps, READY only) ·
  `POST /api/v1/drafts/{id}/regenerate` · `DELETE /api/v1/drafts/{id}`
- Templates : CRUD `GET/POST /api/v1/templates`, `GET/PATCH/DELETE /api/v1/templates/{id}`
- `PATCH /api/v1/profile` étendu (bio, specialties, signature)
- Read models dédiés (`DraftView`/`TemplateView`, ports fail-closed tenant, ADR-0013).

## 6. Frontend

- **Fiche piste** : section « Brouillons » (liste + statut), bouton « Générer un message »
  (modale : type, langue cible — défaut = langue de la paire de la piste —, gabarit optionnel),
  éditeur du brouillon READY (sujet + corps, sauvegarde, **Copier**, Régénérer, Supprimer).
- **Page « Modèles »** (`/templates`, dernière entrée de nav) : liste + édition des gabarits.
- **Profil** : la modale d'objectif devient une petite page/section réglages (bio,
  spécialités, signature, objectif hebdo).
- i18n FR/EN, toasts, confirmation avant suppression ; E2E avec l'adaptateur canned.

## 7. Micro-décisions à valider

1. **Modèle par défaut : Sonnet** (~2 c€/génération, meilleure plume pour des lettres de
   candidature) plutôt que Haiku (~0,5 c€) — surchargeable par env ? *(recommandé : oui)*
2. **Génération asynchrone** (worker + statut GENERATING/READY/FAILED, pattern M1.2) ?
   *(recommandé : oui — pas de requête HTTP bloquée 5-15 s)*
3. Profil M1.4 = **bio + spécialités + signature** ; les **tarifs** attendent un vrai besoin
   (négociation assistée = Futur) ? *(recommandé : oui)*
4. **Canned par défaut sans clé API** (dev/CI/E2E gratuits, bascule Claude par simple env) ?
   *(recommandé : oui)*
5. Sortie M1 = **bouton Copier** (presse-papier), pas d'envoi ? *(recommandé : oui — M2)*
6. **3 gabarits seedés** à la première utilisation + CRUD simple ? *(recommandé : oui)*
